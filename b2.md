# Bài 2

**Nội dung Prompt sau khi tối ưu:**

```text
Hãy đóng vai trò là một Solution Architect và Java Backend Engineer cho hệ thống Notification Service gửi email/SMS cho 1 triệu người dùng.

Nhiệm vụ:
1. Đề xuất ít nhất 3 phương án kiến trúc bất đồng bộ để xử lý gửi thông báo.
2. Lập bảng so sánh trade-offs theo các tiêu chí: độ trễ, khả năng chịu tải, độ tin cậy, retry, chi phí vận hành, độ phức tạp triển khai.
3. Phân tích what-if: nếu traffic tăng gấp 10 lần, nếu worker chết giữa chừng, nếu cần chống gửi trùng message, nếu queue bị nghẽn.
4. Chỉ sau khi so sánh xong mới chọn phương án khuyến nghị cho Spring Boot.
5. Nếu cần code, hãy viết Java ngắn gọn minh họa phương án được chọn.

Ràng buộc:
- Không được mặc định chọn @Async ngay từ đầu.
- Phải cân nhắc Message Queue, Event-driven, Worker-based, hoặc Redis/DB queue.
- Phân tích rõ giả định và rủi ro khi tải cao.
```

**Mẫu bảng so sánh do AI sinh ra:**

```markdown
| Phương án             | Ưu điểm                            | Nhược điểm                           | Khi phù hợp  |
| --------------------- | ---------------------------------- | ------------------------------------ | ------------ |
| @Async                | Dễ làm, ít code                    | Không bền khi app restart, khó scale | Tải nhỏ      |
| Message Queue         | Retry tốt, tách tải, scale tốt     | Cần thêm hạ tầng                     | Tải lớn      |
| Event-driven + Worker | Linh hoạt, dễ mở rộng              | Phức tạp vận hành                    | Hệ thống lớn |
| Redis/DB Queue        | Triển khai đơn giản hơn broker lớn | Dễ nghẽn nếu tải quá cao             | Trung bình   |
```

**Kết luận ngắn:**

- Với 1 triệu người dùng, kiến trúc phù hợp thường là Message Queue + Worker + Retry + Dead Letter Queue.
- @Async chỉ hợp demo hoặc tải nhỏ, không nên là lựa chọn mặc định.
